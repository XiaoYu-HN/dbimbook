Advisor Objectives

Hover your mouse over any blue, italic header for information on this report's tables and columns.

The In-Memory database feature can be employed to improve the performance of a variety of database operations, the greatest of which is analytics processing.

The objective of the In-Memory Advisor is to produce recommendations to optimize analytics processing. As with SQL optimizer cost estimates, estimates for In-Memory sizes and performance benefits need not be precise in order to produce optimal In-Memory configurations.

Workload Database Usage

The In-Memory Advisor measures the activity on objects and SQL statements using Active Session History (ASH) and optimizes for analytics processing. The Advisor differentiates analytics processing from non-analytics (transaction) processing based upon the types of Active Session History (ASH) activity samples and SQL plan information.
The total database time in seconds is the total database activity during the statistics capture window captured from ASH in the Automatic Workload Repository (AWR) and optionally from live ASH data. When using live ASH data, the value will differ from other database reporting facilities.
Analytics processing seconds is the amount of database activity in seconds that is analytical in nature.
Analytics processing percentage is the portion of database activity that is analytical in nature.

Advisor Methods

This table describes how the In-Memory Advisor forms its estimates in order to optimize for In-Memory placement.

The In-Memory Advisor differentiates Analytics Processing from other database activity based upon the SQL plan cardinaility, the Active Session History (ASH) state, employment of parallel query, and other data and statistics.

The In-Memory Advisor estimates the In-Memory size of objects based upon heuristic compression factors along with object data such as row counts, avererage row sizes and column data types.

The In-Memory Advisor estimates Analytic Processing performance improvement factors based upon the following:

  • Elimination of user I/O waits, cluster transfer waits, buffer cache latch waits, etc.

  • Certain query processing advantages related to specific compression types.

  • Decompression cost heuristics per specific compression types.

  • SQL plan selectivity, number of columns in the result set, etc.
